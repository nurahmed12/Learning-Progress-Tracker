ROUND(Cell or Value, number of digits) 
ROUND(E11, 0) Means non decimal round, Rounds to the nearest whole number, e.g., 10.3 > 10 and 10.7 > 11, if 10.5 > 11
ROUND(E11, 1 or any positive number) Means decimal round, Rounds the (1 or any positive number)th value to it's nearest whole number, e.g., ROUND(7.445, 1) > 7.4, ROUND(7.445) > 7.45
ROUND(E11, -1 or any negative number) Means tens, hundreds or more round, Rounds the (1 or any negative number) + 1th number from right most to it's nearest whole number based on the (1 or any negative number)th number, e.g., ROUND(123412, -1) > 123410, ROUND(123416, -1) > 123420, ROUND(123415, -2) > 123400
ROUNDUP(E11, 0 or Positive or Negative) Similar to ROUND() function but it only rounds to the next whole number, e.g., ROUNDUP(5.1, 0) > 6
ROUNDDOWN(E11, 0 or Positive or Negative) Opposite of ROUNDUP()
MROUND(E11, multiple) Rounds to the nearest Multiple, e.g., MROUND(12311, 4) > 12312 Means 12312 is the nearest to 12311 which can be devided by 4

- Editing Group (Sorting):

In order to generate random series you can use the formula =ROUND(Value1, (Value2 - Value1) * RAND(), 2) , this will generate random numbers between Value1 and Value2
1. RAND()
Returns a random decimal number between 0 and 1.
Example output: 0.2438, 0.8793, etc.
2. (b - a)
This is the range or width of the interval.
For example, if a = 5 and b = 10, then (b - a) = 5.
3. (b - a) * RAND()
Scales the random number into your desired range.
Now you get a random decimal between 0 and (b - a).
Example: If RAND() = 0.3, and (b - a) = 5, then this part is 1.5.
4. a + ...
This shifts the number up to start at a.
So you now get a number between a and b.
Example: a = 5, so 5 + 1.5 = 6.5
5. ROUND(..., 2)
Rounds the final result to 2 decimal places.

1. Formula Pattern
   =COUNTIF(<lookup_range>, <this_cell>) = 1
   ‚Ä¢ Returns TRUE if the value in <this_cell> appears exactly once in <lookup_range>.

2. Locking the Lookup Range
   ‚Ä¢ Fixed block (e.g. rows 10‚Äì17 in column‚ÄØK):
     =COUNTIF($K$10:$K$17, K10)=1
     ‚Äì $K$10:$K$17 stays the same when you copy down.
     ‚Äì K10 (no $) changes to K11, K12, etc.

   ‚Ä¢ Entire column:
     =COUNTIF($K:$K, K10)=1
     ‚Äì $K:$K always means the whole column‚ÄØK.
     ‚Äì Good for growing lists; slower on very large sheets; counts blanks too.

3. Criterion Cell Reference
   ‚Ä¢ Relative row (K10): adjusts to each row.
   ‚Ä¢ Absolute row ($K$10): always tests cell‚ÄØK10.

4. ‚Äú$‚Äù Cheat‚ÄëSheet
   ‚Ä¢ $ before column (e.g. $K) locks column.
   ‚Ä¢ $ before row (e.g. K$10) locks row.
   ‚Ä¢ $K$10 locks both column and row (single cell).
   ‚Ä¢ $K:$K locks entire column‚ÄØK.
   ‚Ä¢ $K$10:$K$17 locks that exact block.

CountIF function:

Quick Guide to COUNTIF in Excel

1. Function Structure  
   COUNTIF(range, criteria)  
   ‚Ä¢ Counts how many cells in‚ÄØrange meet the given criteria.

2. Parameters  
   ‚Ä¢ range: the block of cells to check (e.g. A1:A100, K:K)  
   ‚Ä¢ criteria: condition each cell is tested against (number, expression, text, wildcard, or cell reference)

3. Types of Criteria  
   ‚Ä¢ Number  
     ‚Äì COUNTIF(A1:A10, 5)  
       Counts cells exactly equal to 5.  
   ‚Ä¢ Expression (comparison)  
     ‚Äì COUNTIF(B1:B20, ‚Äú>100‚Äù)  
       Counts cells with values greater than 100.  
     ‚Äì Operators: =, <>, >, <, >=, <= (always in quotes)  
   ‚Ä¢ Text  
     ‚Äì COUNTIF(C1:C50, ‚ÄúApple‚Äù)  
       Counts cells exactly matching ‚ÄúApple‚Äù (case‚Äëinsensitive).  
   ‚Ä¢ Wildcards  
     ‚Äì `*` matches any sequence of characters  
       ‚Ä¢ COUNTIF(NameRange, "East*") finds ‚ÄúEast‚Äù, ‚ÄúEastern Campus‚Äù, etc.  
       ‚Ä¢ COUNTIF(NameRange, "*son") finds ‚ÄúJohnson‚Äù, ‚ÄúAnderson‚Äù, etc.  
     ‚Äì `?` matches exactly one character  
       ‚Ä¢ COUNTIF(CodeRange, "A?3") finds ‚ÄúAB3‚Äù, ‚ÄúA63‚Äù but not ‚ÄúA123‚Äù.  
       ‚Ä¢ COUNTIF(WordRange, "?at") finds ‚ÄúCat‚Äù, ‚ÄúBat‚Äù but not ‚ÄúThat‚Äù.  
   ‚Ä¢ Cell reference  
     ‚Äì COUNTIF(F1:F50, G1)  
       Counts cells equal to the value in G1.  
   ‚Ä¢ Expression with reference  
     ‚Äì COUNTIF(F1:F50, ‚Äú>‚Äù & G1)  
       Counts cells greater than the value in G1.  
   ‚Ä¢ Dates  
     ‚Äì COUNTIF(H1:H100, ‚Äú>‚Äù & DATE(2025,1,1))  
       Counts dates after Jan‚ÄØ1,‚ÄØ2025.  
     ‚Äì Can also reference a cell: ‚Äú>‚Äù & A1 where A1 holds a date.

4. Special Behaviors  
   ‚Ä¢ Blanks vs. non‚Äëblanks  
     ‚Äì COUNTIF(A1:A10, ‚Äú‚Äù) counts empty cells.  
     ‚Äì COUNTIF(A1:A10, ‚Äú<>‚Äù) counts cells that are not empty.  
   ‚Ä¢ Mixed data types  
     ‚Äì Numbers in quotes (e.g. ‚Äú5‚Äù) are treated as text, not numeric.  
   ‚Ä¢ Logical values and errors  
     ‚Äì COUNTIF will ignore TRUE, FALSE, and error values unless they match the criteria exactly.  
   ‚Ä¢ Whole‚Äëcolumn ranges (e.g. K:K)  
     ‚Äì Scans all 1,048,576 rows; includes blank cells in the count.  
     ‚Äì Use with care on very large sheets (performance).

5. Common Uses  
   ‚Ä¢ Count specific status  
     ‚Äì COUNTIF(StatusRange, ‚ÄúCompleted‚Äù)  
   ‚Ä¢ Count above/below thresholds  
     ‚Äì COUNTIF(ScoreRange, ‚Äú>=80‚Äù)  
     ‚Äì COUNTIF(ScoreRange, ‚Äú<50‚Äù)  
   ‚Ä¢ Partial text matching  
     ‚Äì COUNTIF(NameRange, ‚ÄúMc*‚Äù) finds names starting with ‚ÄúMc‚Äù.  
   ‚Ä¢ Compare to another cell  
     ‚Äì COUNTIF(DateRange, ‚Äú<‚Äù & DeadlineCell)  
   ‚Ä¢ Unique‚Äëvalue test  
     ‚Äì COUNTIF($K:$K, K10)=1 returns TRUE if K10 is unique in column‚ÄØK.

6. Locking References  
   ‚Ä¢ $A$1:$A$10 locks both column & row (never shifts)  
   ‚Ä¢ A$1 locks row only (column shifts when copying across)  
   ‚Ä¢ $A1 locks column only (row shifts when copying down)  
   ‚Ä¢ $K:$K locks entire column‚ÄØK  
   ‚Ä¢ $K$10:$K$17 locks block K10‚ÄìK17  

7. Relative vs. Absolute Ranges

   ‚Ä¢ For OR logic, combine multiple COUNTIFs or use SUMPRODUCT

   ‚Ä¢ Keep ranges tight (avoid entire columns) for better performance.

Quick Guide: Introduction to Pivot Tables in Excel

1. What Is a Pivot Table?  
   ‚Ä¢ An interactive summary tool that automatically aggregates, analyzes, and presents large data sets.  
   ‚Ä¢ Lets you ‚Äúpivot‚Äù (re‚Äëarrange) rows and columns to see different perspectives.

   ‚Ä¢ You have a detailed table of records (e.g., sales transactions, survey responses).  
   ‚Ä¢ You need quick totals, averages, counts, or subgroup analyses by category.  
   ‚Ä¢ You want an easy drag‚Äëand‚Äëdrop interface instead of complex formulas.

   ‚Ä¢ **Rows area**: Fields you want listed down the side (e.g., Salesman, District).

   ‚Ä¢ Right‚Äëclick a Value and choose **Value Field Settings** to switch from Sum to Count, Average, Min, Max, etc.

   ‚Ä¢ **Sales by month and district**: Rows = Month; Columns = District; Values = Sum of Sales TK.

   ‚Ä¢ Source data should have no blank header names and no completely blank rows/columns.

Logical Functions ‚Äì IF, AND, OR, NOT, IFS

1. IF  
   ‚Ä¢ Structure:  IF(logical_test, value_if_true, value_if_false)  
   ‚Ä¢ Tests one condition and returns one of two results.  
     ‚Äì Example:  =IF(A1>100, "High", "Low")  
       ‚Ä¢ If A1 > 100 returns ‚ÄúHigh,‚Äù otherwise ‚ÄúLow.‚Äù  
   ‚Ä¢ Nesting IFs (pre‚Äë2021):  
     ‚Äì =IF(A1>100, "High", IF(A1>50, "Medium", "Low"))  
     ‚Äì Can become complex with many levels‚Äîconsider IFS.

2. AND, OR, NOT  
   ‚Ä¢ AND(condition1, condition2, ‚Ä¶)  
     ‚Äì Returns TRUE only if ALL conditions are TRUE.  
     ‚Äì Example: =AND(A1>0, B1<10)  
   ‚Ä¢ OR(condition1, condition2, ‚Ä¶)  
     ‚Äì Returns TRUE if ANY condition is TRUE.  
     ‚Äì Example: =OR(A1="Yes", B1="Yes")  
   ‚Ä¢ NOT(condition)  
     ‚Äì Reverses a logical value: TRUE ‚Üí FALSE, FALSE ‚Üí TRUE.  
     ‚Äì Example: =NOT(C1="Complete")  (TRUE if C1 ‚â† ‚ÄúComplete‚Äù)

3. IFS (Excel‚ÄØ2016+ / 365)  
   ‚Ä¢ Structure:  IFS(test1, value1, test2, value2, ‚Ä¶, [TRUE, default])  
   ‚Ä¢ Evaluates tests in order and returns the first matching value.  
     ‚Äì Example: =IFS(A1>100, "High", A1>50, "Medium", A1>=0, "Low", TRUE, "N/A")  
   ‚Ä¢ No need to nest multiple IFs; cleaner and easier to read.

4. Combining with Other Functions  
   ‚Ä¢ IF + AND/OR for multi‚Äëcondition logic:  
     ‚Äì =IF(AND(A1>0, B1<5), "Valid", "Invalid")  
   ‚Ä¢ IFERROR to catch errors:  
     ‚Äì =IFERROR(1/0, "Error!") returns ‚ÄúError!‚Äù instead of #DIV/0!  
   ‚Ä¢ Use logical functions in array formulas for bulk logical tests (Office‚ÄØ365).

5. Common Pitfalls & Tips  
   ‚Ä¢ Blank vs. zero: ISBLANK(A1) or A1="" to test emptiness.  
   ‚Ä¢ Comparisons are case‚Äëinsensitive: "Yes" = "yes".  
   ‚Ä¢ Order matters in IFS‚Äîplace most specific tests first.  
   ‚Ä¢ Avoid deep nesting; switch to IFS or lookup tables (VLOOKUP / XLOOKUP) for complex branching.  
   ‚Ä¢ Always include a default catch‚Äëall (e.g. TRUE, "Other") as the last IFS pair.  

Lookup & Reference ‚Äì VLOOKUP, HLOOKUP, INDEX, MATCH, XLOOKUP

1. VLOOKUP  
   ‚Ä¢ Structure:  VLOOKUP(lookup_value, table_array, col_index, [range_lookup])  
   ‚Ä¢ lookup_value: value to find in first column of table_array  
   ‚Ä¢ table_array: block of cells (e.g. A1:D100)  
   ‚Ä¢ col_index: column number to return (1 = first column)  
   ‚Ä¢ range_lookup: FALSE for exact match, TRUE (or omitted) for approximate  
   ‚Ä¢ Example: =VLOOKUP(B2, Products!$A$2:$C$100, 3, FALSE)  
   ‚Ä¢ Pitfalls:  
     ‚Äì Only searches left‚Äëto‚Äëright  
     ‚Äì Requires first column sorted for approximate matches  
     ‚Äì Exact match must use FALSE  

2. HLOOKUP  
   ‚Ä¢ Structure:  HLOOKUP(lookup_value, table_array, row_index, [range_lookup])  
   ‚Ä¢ Same logic as VLOOKUP but searches in first row, returns from row_index  
   ‚Ä¢ Example: =HLOOKUP("Jan", Sales!$A$1:$M$5, 4, FALSE)  
   ‚Ä¢ Pitfalls:  
     ‚Äì Only top‚Äëto‚Äëbottom  
     ‚Äì Range must be sorted for TRUE  

3. INDEX  
   ‚Ä¢ Two forms:  
     ‚Äì Array form: INDEX(array, row_num, [column_num])  
     ‚Äì Reference form: INDEX(reference, row_num, [col_num], [area_num])  
   ‚Ä¢ Returns value at intersection of row_num & column_num  
   ‚Ä¢ Example: =INDEX(A2:C10, 5, 2) returns value in row‚ÄØ5, column‚ÄØ2 of that range  
   ‚Ä¢ Use with MATCH for flexible lookups  

4. MATCH  
   ‚Ä¢ Structure:  MATCH(lookup_value, lookup_array, [match_type])  
   ‚Ä¢ lookup_array: single row or column  
   ‚Ä¢ match_type: 0 for exact, 1 for <= (sorted ascending), -1 for >= (sorted descending)  
   ‚Ä¢ Returns position (number) of found item in lookup_array  
   ‚Ä¢ Example: =MATCH("Widget X", A2:A50, 0) returns the row offset  

5. INDEX + MATCH  
   ‚Ä¢ Combines strengths:  
     ‚Äì Lookup in any column/row, not limited to first  
     ‚Äì No need to sort for exact matches  
   ‚Ä¢ Example: =INDEX(C2:C50, MATCH(E2, A2:A50, 0))  
     ‚Äì Finds E2 in A2:A50, returns corresponding C column value  

6. XLOOKUP (Excel‚ÄØ365/2021+)  
   ‚Ä¢ Structure: XLOOKUP(lookup_value, lookup_array, return_array, [if_not_found], [match_mode], [search_mode])  
   ‚Ä¢ if_not_found: value when no match (default #N/A)  
   ‚Ä¢ match_mode: 0 exact, -1 exact or next smaller, 1 exact or next larger, 2 wildcard  
   ‚Ä¢ search_mode: 1 first‚Äëto‚Äëlast, -1 last‚Äëto‚Äëfirst  
   ‚Ä¢ Example: =XLOOKUP(E2, A2:A50, C2:C50, "Not found", 0, 1)  
   ‚Ä¢ Advantages:  
     ‚Äì Lookup left or right, vertical or horizontal  
     ‚Äì Built‚Äëin error handling  
     ‚Äì Supports wildcards and reverse search  

7. Common Tips & Pitfalls  
   ‚Ä¢ Always lock ranges with $ when copying formulas  
   ‚Ä¢ Use exact match (FALSE or 0) to avoid unexpected results  
   ‚Ä¢ For large data sets, INDEX/MATCH or XLOOKUP are faster than VLOOKUP  
   ‚Ä¢ XLOOKUP replaces most legacy lookups‚Äîuse it when available

Text Functions ‚Äì CONCATENATE, LEFT, RIGHT, MID, TRIM, LEN, SUBSTITUTE

1. CONCATENATE (or CONCAT in newer Excel)  
   ‚Ä¢ Structure: CONCATENATE(text1, text2, ‚Ä¶)  
   ‚Ä¢ Joins multiple text strings into one.  
     ‚Äì Example: =CONCATENATE(A1, " ", B1) merges first and last name with a space.  
   ‚Ä¢ Newer: =CONCAT(A1, " ", B1)  
   ‚Ä¢ TIP: Use TEXTJOIN(delimiter, ignore_empty, range1, ‚Ä¶) to join ranges with a delimiter and skip blanks.

2. LEFT  
   ‚Ä¢ Structure: LEFT(text, [num_chars])  
   ‚Ä¢ Returns the leftmost characters from a string.  
     ‚Äì Example: =LEFT("Report2025", 6) ‚Üí "Report"  
   ‚Ä¢ num_chars defaults to 1 if omitted.  
   ‚Ä¢ If num_chars > length, returns the entire string.

3. RIGHT  
   ‚Ä¢ Structure: RIGHT(text, [num_chars])  
   ‚Ä¢ Returns the rightmost characters.  
     ‚Äì Example: =RIGHT("Report2025", 4) ‚Üí "2025"  
   ‚Ä¢ num_chars defaults to 1.  
   ‚Ä¢ If num_chars > length, returns the entire string.

4. MID  
   ‚Ä¢ Structure: MID(text, start_num, num_chars)  
   ‚Ä¢ Extracts a substring starting at a position.  
     ‚Äì Example: =MID("Spreadsheet", 6, 5) ‚Üí "sheet" (starts at 6th character, length 5)  
   ‚Ä¢ If start_num > length, returns "".  
   ‚Ä¢ If start_num + num_chars exceeds length, returns up to the end.

5. TRIM  
   ‚Ä¢ Structure: TRIM(text)  
   ‚Ä¢ Removes extra spaces: leading, trailing, and reduces multiple spaces between words to one.  
     ‚Äì Example: =TRIM("  Hello   World  ") ‚Üí "Hello World"  
   ‚Ä¢ Does NOT remove non‚Äëbreaking spaces (use CLEAN for non‚Äëprintables).

6. LEN  
   ‚Ä¢ Structure: LEN(text)  
   ‚Ä¢ Returns the number of characters in a string, including spaces.  
     ‚Äì Example: =LEN("Excel Tips") ‚Üí 10  
   ‚Ä¢ Useful to pair with LEFT/RIGHT/MID to calculate dynamic lengths.

7. SUBSTITUTE  
   ‚Ä¢ Structure: SUBSTITUTE(text, old_text, new_text, [instance_num])  
   ‚Ä¢ Replaces occurrences of old_text with new_text.  
     ‚Äì Example: =SUBSTITUTE("A-B-A-B", "B", "X") ‚Üí "A-X-A-X"  
     ‚Äì To replace only the 2nd instance: =SUBSTITUTE("A-B-A-B","B","X",2) ‚Üí "A-B-A-X"  
   ‚Ä¢ Case‚Äësensitive.  
   ‚Ä¢ For case‚Äëinsensitive, use UPPER/LOWER wrappers.

8. Common Tips & Pitfalls  
   ‚Ä¢ Always wrap cell references in TEXT() when concatenating numbers with formats: =TEXT(A1,"0.00")  
   ‚Ä¢ Remember Excel treats everything as text when joined‚Äîuse VALUE() to convert back to number.  
   ‚Ä¢ Combine TRIM and CLEAN(text) to strip extra spaces and non‚Äëprintable characters before processing.  
   ‚Ä¢ Use LEN in formulas like =LEN(A1)-LEN(SUBSTITUTE(A1,"a","")) to count occurrences of a character.

Excel Custom Date Formats: d, dd, ddd, dddd

Purpose: Control how dates display in Excel by defining day formats (numeric and weekday names).

1. d
   - Displays: Day of month without leading zero.
   - Range: 1‚Äì31
   - Example: For March 5, 2025 ‚Üí 5

2. dd
   - Displays: Day of month with two digits (adds leading zero if needed).
   - Range: 01‚Äì31
   - Example: For March 5, 2025 ‚Üí 05

3. ddd
   - Displays: Abbreviated weekday name.
   - Example: For March 5, 2025 (Wednesday) ‚Üí Wed

4. dddd
   - Displays: Full weekday name.
   - Example: For March 5, 2025 ‚Üí Wednesday

How Excel Interprets Repeated Letters:
- Runs of identical letters form a single instruction (e.g., dddd is one token: full weekday name).
- More than four letters (e.g., ddddd) behaves like four letters (dddd).

Usage Examples:
- `ddd, mmm dd, yyyy` ‚Üí Wed, Mar 05, 2025
- `dddd, mmmm d"th"` ‚Üí Wednesday, March 5th

Tip: Combine tokens (d, m, y) to build custom date displays matching your reporting or presentation needs.

Aggregation Functions ‚Äì SUMIFS, COUNTIFS, AVERAGEIFS

1. Function Structures  
   ‚Ä¢ SUMIFS(sum_range, criteria_range1, criteria1, [criteria_range2, criteria2], ‚Ä¶)  
   ‚Ä¢ COUNTIFS(criteria_range1, criteria1, [criteria_range2, criteria2], ‚Ä¶)  
   ‚Ä¢ AVERAGEIFS(average_range, criteria_range1, criteria1, [criteria_range2, criteria2], ‚Ä¶)

2. Parameters  
   ‚Ä¢ sum_range / average_range: the cells to sum or average (for COUNTIFS omit this)  
   ‚Ä¢ criteria_rangeN: the range to test for each condition  
   ‚Ä¢ criteriaN: the test (number, expression, text, wildcard, or cell ref)

3. Criteria Types  
   ‚Ä¢ Number or exact text: ‚Äú=5‚Äù, ‚ÄúApple‚Äù  
     ‚Äì Example: criteria1 = 5 or "Completed"  
   ‚Ä¢ Comparison operators: ‚Äú>100‚Äù, ‚Äú<=50‚Äù (must be in quotes)  
   ‚Ä¢ Wildcards:  
     ‚Äì ‚ÄúEast*‚Äù for any text starting with ‚ÄúEast‚Äù  
     ‚Äì ‚Äú?at‚Äù for any three‚Äëletter text ending in ‚Äúat‚Äù  
   ‚Ä¢ Cell references: ‚Äú>‚Äù & A1, ‚Äú<>‚Äù & B1  
   ‚Ä¢ Dates: ‚Äú>=‚Äù & DATE(2025,1,1) or ‚Äú<‚Äù & C1 (where C1 holds a date)

4. Examples  
   ‚Ä¢ Sum sales in North region in July:  
     =SUMIFS(SalesRange, RegionRange, "North", MonthRange, "July")  
   ‚Ä¢ Count orders >100 and status = ‚ÄúShipped‚Äù:  
     =COUNTIFS(OrderQty, ">100", StatusRange, "Shipped")  
   ‚Ä¢ Average score for group A or B:  
     =AVERAGEIFS(ScoreRange, GroupRange, "A", GroupRange, "B")  
     (Note: to OR groups, use two SUMIFS/COUNTIFS or a filtered subset)

5. Key Behaviors & Notes  
   ‚Ä¢ All ranges must be the same size and shape; mismatched ranges cause #VALUE!  
   ‚Ä¢ COUNTIFS and AVERAGEIFS allow multiple AND conditions only.  
   ‚Ä¢ For OR logic, sum multiple SUMIFS or wrap COUNTIFS in SUM:  
     =SUM(SUMIFS(...Group="A"...), SUMIFS(...Group="B"...))  
   ‚Ä¢ Blank criteria: use ‚Äú‚Äù to test blanks, "<>" to test non‚Äëblanks.  
   ‚Ä¢ Criteria ranges can be whole columns (e.g. A:A), but may impact performance.

6. Locking References  
   ‚Ä¢ Use $A$1:$A$100 to fix ranges when copying formulas  
   ‚Ä¢ Mix $A1 or A$1 to lock only row or column as needed  
   ‚Ä¢ Ensure sum/average ranges stay aligned with criteria ranges

7. Tips & Best Practices  
   ‚Ä¢ Place the most restrictive criteria first for readability  
   ‚Ä¢ Avoid entire‚Äëcolumn ranges on large workbooks to speed up calculations  
   ‚Ä¢ Use named ranges or structured table references for clarity  
   ‚Ä¢ Wrap SUMIFS in IFERROR to handle ‚Äúno match‚Äù cases:  
     =IFERROR(SUMIFS(...), 0)  
   ‚Ä¢ Validate with COUNTIFS that your SUMIFS scope is correct before averaging

Database Concepts ‚Äì Tables, Schemas, Primary/Foreign Keys & Indexes

1. Tables  
   ‚Ä¢ Definition: Structured collection of related data organized in rows (records) and columns (fields).  
   ‚Ä¢ CREATE:  
     ‚Äí CREATE TABLE table_name (  
     ‚Äí ‚ÄÉ column1 datatype [constraints],  
     ‚Äí ‚ÄÉ column2 datatype [constraints],  
     ‚Äí ‚ÄÉ ‚Ä¶  
     ‚Äí );  
   ‚Ä¢ Data Types: INT, VARCHAR(n), DATE, DECIMAL(p,s), BOOLEAN, etc.  
   ‚Ä¢ ALTER: add, modify or drop columns after creation (ALTER TABLE).

2. Schemas  
   ‚Ä¢ Definition: Logical container or namespace for database objects (tables, views, procedures).  
   ‚Ä¢ Purpose: Organize objects, manage permissions, avoid name collisions.  
   ‚Ä¢ CREATE:  
     ‚Äí CREATE SCHEMA schema_name;  
   ‚Ä¢ Usage:  
     ‚Äí schema_name.table_name to qualify cross‚Äëschema references.  
     ‚Äí GRANT/REVOKE permissions at schema level.

3. Primary Keys  
   ‚Ä¢ Definition: Uniquely identifies each row in a table.  
   ‚Ä¢ Characteristics:  
     ‚Äí NOT NULL, UNIQUE, minimal (single or composite).  
   ‚Ä¢ Declaration:  
     ‚Äí Inline: id INT PRIMARY KEY,  
     ‚Äí Or out‚Äëof‚Äëline: PRIMARY KEY (column1, column2).  
   ‚Ä¢ Purpose: Ensures row uniqueness, used by foreign keys and indexes.

4. Foreign Keys  
   ‚Ä¢ Definition: Column(s) in one table that reference primary key(s) of another.  
   ‚Ä¢ Declaration:  
     ‚Äí CREATE TABLE orders (  
     ‚Äí ‚ÄÉ order_id INT PRIMARY KEY,  
     ‚Äí ‚ÄÉ customer_id INT,  
     ‚Äí ‚ÄÉ FOREIGN KEY (customer_id) REFERENCES customers(id)  
     ‚Äí );  
   ‚Ä¢ Options:  
     ‚Äí ON DELETE CASCADE / SET NULL / RESTRICT  
     ‚Äí ON UPDATE CASCADE / SET DEFAULT  
   ‚Ä¢ Purpose: Enforce referential integrity between related tables.

5. Indexes  
   ‚Ä¢ Definition: Data structure that improves query performance on one or more columns.  
   ‚Ä¢ Types:  
     ‚Äí Single‚Äëcolumn: CREATE INDEX idx_name ON table(col);  
     ‚Äí Composite: CREATE INDEX idx_name ON table(col1, col2);  
     ‚Äí Unique: CREATE UNIQUE INDEX idx_name ON table(col);  
     ‚Äí Full‚Äëtext, Spatial (vendor‚Äëspecific).  
   ‚Ä¢ Considerations:  
     ‚Äí Speed up SELECT WHERE, JOIN, ORDER BY.  
     ‚Äí Slow down INSERT/UPDATE/DELETE due to maintenance.  
     ‚Äí Use sparingly on high‚Äëwrite tables.  
   ‚Ä¢ Drop: DROP INDEX idx_name ON table;

6. Best Practices  
   ‚Ä¢ Always define a primary key for every table.  
   ‚Ä¢ Use schemas to group related objects and manage access.  
   ‚Ä¢ Index columns used frequently in WHERE, JOIN, GROUP BY, ORDER BY.  
   ‚Ä¢ Avoid over‚Äëindexing‚Äîmonitor performance and index usage.  
   ‚Ä¢ Name conventions: prefix indexes (idx_), fkeys (fk_), tables plural (users).

SQL Filtering ‚Äì AND, OR, NOT, BETWEEN, IN, LIKE, IS NULL

1. AND  
   ‚Ä¢ Combines multiple conditions; all must be true.  
   ‚Ä¢ Example:  
     SELECT * FROM students

2. OR  
   ‚Ä¢ At least one condition must be true.  
   ‚Ä¢ Example:  
     SELECT * FROM courses  
     WHERE subject = 'Math' OR subject = 'Physics';

3. NOT  
   ‚Ä¢ Reverses a condition‚Äôs result.  
   ‚Ä¢ Example:  
     SELECT * FROM users  
     WHERE NOT status = 'inactive';

4. BETWEEN  
   ‚Ä¢ Checks if a value is within a range (inclusive).  
   ‚Ä¢ Example:  
     SELECT * FROM orders  
     WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31';

5. IN  
   ‚Ä¢ Checks if a value matches any item in a list.  
   ‚Ä¢ Example:  
     SELECT * FROM employees  
     WHERE department IN ('HR', 'IT', 'Finance');  
   ‚Ä¢ Can also use with NOT:  
     WHERE department NOT IN ('Sales', 'Marketing');

6. LIKE  
   ‚Ä¢ Pattern matching using wildcards:  
     ‚Äì `%` = any sequence of characters  
     ‚Äì `_` = any single character  
   ‚Ä¢ Examples:  
     SELECT * FROM customers  
     WHERE name LIKE 'J%';‚ÄÉ‚ÄÉ-- starts with 'J'  
     SELECT * FROM files

Sorting & Limiting ‚Äì ORDER BY, LIMIT, TOP

   ‚Ä¢ Always pair ORDER BY with LIMIT/TOP for consistent results.  
   ‚Ä¢ ORDER BY can use column numbers: ORDER BY 2 DESC (2nd column).

   ‚Ä¢ Avoid using LIMIT without ORDER BY‚Äîit may return unpredictable rows.

Aggregations ‚Äì COUNT, SUM, AVG, MIN, MAX, GROUP¬†BY & HAVING

1. COUNT  
   ‚Ä¢ COUNT(*)  
     ‚Äì Counts all rows, including those with NULLs in other columns.  
   ‚Ä¢ COUNT(column)  
     ‚Äì Counts only non‚ÄëNULL values in that column.  
   ‚Ä¢ COUNT(DISTINCT column)

   ‚Ä¢ Example:

   ‚Ä¢ SUM(numeric_column)

   ‚Ä¢ Returns NULL if no non‚ÄëNULL rows (use COALESCE to default to 0).  
   ‚Ä¢ Example:

   ‚Ä¢ AVG(numeric_column)

   ‚Ä¢ Ignores NULLs automatically.  
   ‚Ä¢ Example:

   ‚Ä¢ MIN(column)

   ‚Ä¢ Groups rows sharing the same values in listed columns.

   ‚Ä¢ Example:

   ‚Ä¢ Filters groups after aggregation (WHERE cannot use aggregates).  
   ‚Ä¢ Syntax:

   ‚Ä¢ Example:  
     SELECT department, COUNT(*) AS num_emps

Joins ‚Äì INNER, LEFT, RIGHT, FULL¬†OUTER & CROSS

1. INNER JOIN  
   ‚Ä¢ Returns rows where the join condition matches in both tables.  
   ‚Ä¢ Syntax:  
     SELECT a.col1, b.col2  
     FROM tableA AS a  
     INNER JOIN tableB AS b  
       ON a.key = b.key;  
   ‚Ä¢ Example:  
     SELECT e.name, d.dept_name  
     FROM employees e  
     INNER JOIN departments d  
       ON e.dept_id = d.id;  
   ‚Ä¢ Only matching pairs appear in results.

2. LEFT JOIN (or LEFT OUTER JOIN)  
   ‚Ä¢ Returns all rows from the left table, plus matching rows from the right.  
   ‚Ä¢ Non‚Äëmatching right rows yield NULLs.  
   ‚Ä¢ Syntax:  
     SELECT a.*, b.*  
     FROM tableA a  
     LEFT JOIN tableB b  
       ON a.key = b.key;  
   ‚Ä¢ Example:  
     SELECT c.customer, o.order_id  
     FROM customers c  
     LEFT JOIN orders o  
       ON c.id = o.customer_id;  
   ‚Ä¢ Use to find ‚Äúorphan‚Äù left‚Äëside records with no match.

3. RIGHT JOIN (or RIGHT OUTER JOIN)  
   ‚Ä¢ Returns all rows from the right table, plus matching rows from the left.  
   ‚Ä¢ Non‚Äëmatching left rows yield NULLs.  
   ‚Ä¢ Syntax:  
     SELECT a.*, b.*  
     FROM tableA a  
     RIGHT JOIN tableB b  
       ON a.key = b.key;  
   ‚Ä¢ Example:  
     SELECT p.product, s.sale_id  
     FROM products p  
     RIGHT JOIN sales s  
       ON p.id = s.product_id;  
   ‚Ä¢ Useful when focusing on the right table‚Äôs completeness.

4. FULL OUTER JOIN  
   ‚Ä¢ Returns all rows when there is a match in either table.  
   ‚Ä¢ Non‚Äëmatching columns from either side yield NULLs.  
   ‚Ä¢ Syntax:  
     SELECT a.*, b.*  
     FROM tableA a  
     FULL OUTER JOIN tableB b  
       ON a.key = b.key;  
   ‚Ä¢ Example:  
     SELECT e.name, m.manager_name  
     FROM employees e  
     FULL OUTER JOIN managers m  
       ON e.manager_id = m.id;  
   ‚Ä¢ Combines LEFT and RIGHT JOIN; shows all records.

5. CROSS JOIN  
   ‚Ä¢ Returns the Cartesian product of both tables (every combination).  
   ‚Ä¢ Syntax:  
     SELECT a.col, b.col  
     FROM tableA a  
     CROSS JOIN tableB b;  
   ‚Ä¢ Example:  
     SELECT p.product, d.discount_code  
     FROM products p  
     CROSS JOIN discounts d;  
   ‚Ä¢ Use sparingly‚Äîresult size = rowsA √ó rowsB.

6. JOIN Tips & Best Practices  
   ‚Ä¢ Always qualify column names when joining (e.g., a.id, b.id) to avoid ambiguity.  
   ‚Ä¢ Prefer explicit JOIN syntax over comma‚Äëseparated tables for clarity.  
   ‚Ä¢ Index join columns to improve performance.  
   ‚Ä¢ Use INNER JOIN for filtering to matching records only; OUTER JOINs to retain unmatched rows.  
   ‚Ä¢ For FULL OUTER in MySQL (which lacks it), simulate with UNION of LEFT and RIGHT JOINs.  

Set Operations ‚Äì UNION, UNION ALL, INTERSECT & EXCEPT

1. UNION  
   ‚Ä¢ Combines results of two queries into one result set, removing duplicates.  
   ‚Ä¢ Syntax:  
     SELECT col1, col2, ‚Ä¶  
     FROM tableA  
     UNION  
     SELECT col1, col2, ‚Ä¶  
     FROM tableB;  
   ‚Ä¢ Requirements: same number of columns and compatible data types in each SELECT.

2. UNION ALL  
   ‚Ä¢ Same as UNION but **keeps** duplicate rows.  
   ‚Ä¢ Syntax:  
     SELECT ‚Ä¶ FROM tableA  
     UNION ALL  
     SELECT ‚Ä¶ FROM tableB;  
   ‚Ä¢ Faster than UNION because it skips the duplicate elimination step.

3. INTERSECT  
   ‚Ä¢ Returns only rows **common** to both queries.  
   ‚Ä¢ Syntax (if supported):  
     SELECT ‚Ä¶ FROM tableA  
     INTERSECT  
     SELECT ‚Ä¶ FROM tableB;  
   ‚Ä¢ Eliminates duplicates in the result.

4. EXCEPT (or MINUS)  
   ‚Ä¢ Returns rows from the first query that **don‚Äôt** appear in the second.  
   ‚Ä¢ Syntax (SQL Server/PostgreSQL):  
     SELECT ‚Ä¶ FROM tableA

     SELECT ‚Ä¶ FROM tableB;  
   ‚Ä¢ Oracle uses MINUS instead of EXCEPT.  
   ‚Ä¢ Eliminates duplicates in the result.

   ‚Ä¢ Column order and types must match across all SELECTs.  
   ‚Ä¢ ORDER BY can only appear at the **end** of the final SELECT.  
   ‚Ä¢ Use UNION ALL when you need full combined data (including duplicates).  
   ‚Ä¢ INTERSECT & EXCEPT require database support (not in MySQL without workaround).  
   ‚Ä¢ Performance: set operations can trigger sorting/hash operations‚Äîindex underlying tables where possible.

Window Functions ‚Äì ROW_NUMBER, RANK, DENSE_RANK, LEAD & LAG

1. ROW_NUMBER()  
   ‚Ä¢ Assigns a unique sequential integer to each row within a partition, based on the ORDER BY clause.  
   ‚Ä¢ Syntax:  
     ROW_NUMBER() OVER (PARTITION BY partition_expr ORDER BY sort_expr [ASC|DESC])  
   ‚Ä¢ Example:  
     SELECT name, department,  
            ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank  
     FROM employees;  
   ‚Ä¢ Use for ‚Äútop‚ÄëN per group‚Äù queries or to generate unique row IDs.

2. RANK()  
   ‚Ä¢ Assigns a ranking number to rows within a partition; ties receive the same rank, but gaps appear after ties.  
   ‚Ä¢ Syntax:  
     RANK() OVER (PARTITION BY partition_expr ORDER BY sort_expr)  
   ‚Ä¢ Example:  
     SELECT sales_rep, region,  
            RANK() OVER (PARTITION BY region ORDER BY sales DESC) AS region_rank  
     FROM sales;  
   ‚Ä¢ If two rows tie for rank 1, the next rank is 3.

3. DENSE_RANK()  
   ‚Ä¢ Similar to RANK(), but ranks are consecutive‚Äîno gaps after ties.  
   ‚Ä¢ Syntax:  
     DENSE_RANK() OVER (PARTITION BY partition_expr ORDER BY sort_expr)  
   ‚Ä¢ Example:  
     SELECT product, category,  
            DENSE_RANK() OVER (PARTITION BY category ORDER BY units_sold DESC) AS category_rank  
     FROM inventory;  
   ‚Ä¢ If two items tie for rank 1, the next rank is 2.

4. LEAD()  
   ‚Ä¢ Accesses data from the next row (offset forward) within the partition.  
   ‚Ä¢ Syntax:  
     LEAD(value_expr, offset[, default]) OVER (PARTITION BY partition_expr ORDER BY sort_expr)  
   ‚Ä¢ Example:  
     SELECT date, revenue,  
            LEAD(revenue, 1, 0) OVER (ORDER BY date) AS next_day_revenue  
     FROM daily_sales;  
   ‚Ä¢ Use to compare current vs. next row values (e.g., trend analysis).

5. LAG()  
   ‚Ä¢ Accesses data from the previous row (offset backward) within the partition.  
   ‚Ä¢ Syntax:  
     LAG(value_expr, offset[, default]) OVER (PARTITION BY partition_expr ORDER BY sort_expr)  
   ‚Ä¢ Example:  
     SELECT date, revenue,  
            LAG(revenue, 1, 0) OVER (ORDER BY date) AS prev_day_revenue  
     FROM daily_sales;  
   ‚Ä¢ Use to compute differences between current and prior rows.

6. Common Notes  
   ‚Ä¢ All window functions require an OVER(...) clause.  
   ‚Ä¢ PARTITION BY defines groups; omit it to treat the entire result set as one partition.  
   ‚Ä¢ ORDER BY inside OVER controls the sequence for ranking or lead/lag.  
   ‚Ä¢ Window functions do not reduce row count‚Äîthey return one result per input row.  
   ‚Ä¢ Combine with a WHERE or CTE to filter for specific ranks or trend conditions.  

Subqueries: Correlated & Non-Correlated

1. What is a Subquery?  
   ‚Ä¢ A subquery is a query nested inside another query (SELECT, FROM, or WHERE).  
   ‚Ä¢ It provides intermediate results used by the outer/main query.

‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
2. Non-Correlated Subquery  
   ‚Ä¢ Executes independently of the outer query.  
   ‚Ä¢ Runs once and returns a single or multiple values.  
   ‚Ä¢ Commonly used in WHERE, FROM, or SELECT clauses.

   Example 1: Scalar subquery (returns one value)  
   SELECT name  
   FROM employees  
   WHERE salary > (SELECT AVG(salary) FROM employees);  

   Example 2: IN subquery (returns multiple values)  
   SELECT name  
   FROM employees  
   WHERE department_id IN (SELECT id FROM departments WHERE location = 'New York');  

   Characteristics:  
   - Inner query runs first.  
   - No dependency on outer query‚Äôs row.  
   - More performant than correlated subqueries.

‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
3. Correlated Subquery  
   ‚Ä¢ Depends on each row of the outer query to execute.  
   ‚Ä¢ Executes once per row of the outer query.  
   ‚Ä¢ Typically used in WHERE or SELECT clauses.

   Example:  
   SELECT e.name  
   FROM employees e  
   WHERE salary > (SELECT AVG(salary)  
                   FROM employees  
                   WHERE department_id = e.department_id);  

   Characteristics:  
   - Inner query refers to outer query‚Äôs columns (e.department_id).  
   - Slower for large data sets due to repeated execution.  
   - Useful for row-by-row comparisons (e.g., rank within group).

4. Subquery in FROM Clause (Inline View)  
   ‚Ä¢ Acts like a temporary table or view.  
   ‚Ä¢ Can be joined with other tables.

   Example:  
   SELECT dept, avg_sal  
   FROM (SELECT department_id AS dept, AVG(salary) AS avg_sal  
         FROM employees  
         GROUP BY department_id) AS dept_avg  
   WHERE avg_sal > 50000;

5. Subquery in SELECT Clause  
   ‚Ä¢ Useful for embedding a scalar calculation for each row.  

   Example:  
   SELECT name,  
          (SELECT COUNT(*) FROM tasks WHERE tasks.emp_id = e.id) AS task_count  
   FROM employees e;

6. Tips  
   ‚Ä¢ Use correlated subqueries carefully‚Äîthey‚Äôre powerful but can be slow.  
   ‚Ä¢ Always test subqueries independently when debugging.  
   ‚Ä¢ Prefer joins or CTEs if performance matters and logic permits.  

Common Table Expressions (CTEs)

1. What is a CTE?  
   ‚Ä¢ A Common Table Expression (CTE) is a temporary result set that you can reference within a `SELECT`, `INSERT`, `UPDATE`, or `DELETE` statement.  
   ‚Ä¢ It improves readability, modularizes complex queries, and supports recursion.

2. Syntax  
   WITH cte_name AS (  
       SELECT ...  
   )  
   SELECT * FROM cte_name;

   ‚Ä¢ You can define multiple CTEs separated by commas:
     WITH cte1 AS (SELECT ...),  
          cte2 AS (SELECT ... FROM cte1)  
     SELECT * FROM cte2;

3. Example ‚Äì Basic CTE  
   WITH dept_avg AS (  
       SELECT department_id, AVG(salary) AS avg_salary  
       FROM employees  
       GROUP BY department_id  
   )  
   SELECT e.name, e.salary, d.avg_salary  
   FROM employees e  
   JOIN dept_avg d ON e.department_id = d.department_id;

4. Benefits  
   ‚Ä¢ Improves query clarity by breaking complex logic into parts.  
   ‚Ä¢ Avoids repeating subqueries.  
   ‚Ä¢ Can be referenced multiple times within the main query.  
   ‚Ä¢ Replaces views for temporary operations.

5. Recursive CTE  
   ‚Ä¢ A CTE that calls itself to return hierarchical or sequential data.

   Syntax:  
   WITH RECURSIVE cte_name AS (  
       -- Anchor member  
       SELECT ...  
       UNION ALL  
       -- Recursive member  
       SELECT ... FROM cte_name WHERE condition  
   )  
   SELECT * FROM cte_name;

   Example ‚Äì Numbers 1 to 5:  
   WITH RECURSIVE numbers AS (  
       SELECT 1 AS n  
       UNION ALL  
       SELECT n + 1 FROM numbers WHERE n < 5  
   )  
   SELECT * FROM numbers;

6. CTE vs Subquery  
   | Feature            | CTE                        | Subquery                    |  
   |--------------------|-----------------------------|------------------------------|  
   | Readability        | More readable               | Can be nested & confusing    |  
   | Reusability        | Can reference multiple times| Must repeat for reuse        |  
   | Recursion support  | Yes                         | No                           |  
   | Temporary storage  | Yes                         | Yes                          |  

7. Notes  
   ‚Ä¢ CTE only exists during execution of the query.  
   ‚Ä¢ ORDER BY inside a CTE does not affect final results unless combined with TOP, OFFSET, or used in final SELECT.  
   ‚Ä¢ Recursive CTEs must include a termination condition to avoid infinite loops.

‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
1. Transactions  
‚Ä¢ A transaction is a sequence of one or more SQL operations treated as a single logical unit of work.
‚Ä¢ Ensures data integrity using **ACID** properties.

ACID Properties:
  - **Atomicity**: All steps succeed or none at all.
  - **Consistency**: Keeps the database in a valid state.
  - **Isolation**: Transactions don‚Äôt interfere with each other.
  - **Durability**: Once committed, changes are permanent.

Basic Transaction Commands:
  - `BEGIN` / `START TRANSACTION` ‚Äì Starts a new transaction.
  - `COMMIT` ‚Äì Saves all changes.
  - `ROLLBACK` ‚Äì Undoes changes since the last BEGIN.
  - `SAVEPOINT` ‚Äì Sets a point you can roll back to.
  - `ROLLBACK TO SAVEPOINT` ‚Äì Rolls back to a specific point.

Example:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

If an error occurs, you can use ROLLBACK; to undo all changes.

Use Cases:
- Bank transfers
- Batch updates
- Ensuring data consistency in multi-step operations

2. Basic SQL Optimization  
‚Ä¢ Optimization improves the performance and efficiency of queries.

Tips:

‚úÖ Use SELECT only needed columns  
-- Bad

-- Good  
SELECT name, salary FROM employees;

‚úÖ Use WHERE clauses to reduce scanned rows  
SELECT * FROM orders WHERE status = 'shipped';

‚úÖ Use appropriate indexes  
- Create indexes on columns used in JOIN, WHERE, and ORDER BY.

‚úÖ Avoid functions on indexed columns in WHERE  
-- Slower

-- Faster

‚úÖ Avoid SELECT DISTINCT if not needed  
- DISTINCT is expensive. Only use it when necessary.

‚úÖ Limit rows returned

‚úÖ Use EXISTS instead of IN for subqueries with large datasets  
-- Better

   SELECT 1 FROM orders o WHERE o.customer_id = c.id

‚úÖ Analyze execution plans

Index ‚Äì Detailed SQL Note (Including Storage & Organization Types)

1. What Is an Index?  
‚Ä¢ A database object that speeds up data retrieval by creating a lookup structure on one or more columns.  
‚Ä¢ Analogous to a book‚Äôs index: points directly to rows instead of scanning every page.

2. Index Types by Structure  
‚Ä¢ **Clustered Index**  
  ‚Äì Physically orders the table rows on disk to match the index key.  
  ‚Äì Only one per table (because rows can be in only one order).  
  ‚Äì Example (SQL Server):  
    CREATE CLUSTERED INDEX idx_emp_id ON employees(employee_id);  
‚Ä¢ **Non-Clustered Index**  
  ‚Äì Separate structure that stores key values and row pointers (RID or clustering key).  
  ‚Äì Can have many per table.  
  ‚Äì Example:  
    CREATE INDEX idx_emp_lastname ON employees(last_name);  
‚Ä¢ **Heap**  
  ‚Äì A table without any clustered index; rows are stored in no particular order.  
  ‚Äì Lookups rely on non-clustered indexes or full scans.

3. Index Types by Storage Model  
‚Ä¢ **Rowstore Index** (traditional B-Tree)  
  ‚Äì Organizes data row by row.  
  ‚Äì Ideal for transactional (OLTP) workloads with frequent single-row lookups.  
‚Ä¢ **Columnstore Index**  
  ‚Äì Stores data column by column in compressed segments.  
  ‚Äì Superior for analytical (OLAP) queries aggregating large data volumes.  
  ‚Äì Example (SQL Server):  
    CREATE COLUMNSTORE INDEX idx_sales_cs ON sales(sale_date, amount);  
‚Ä¢ **Clustered Columnstore**  
  ‚Äì Replaces the base table with a columnstore format‚Äîno separate rowstore table.  
  ‚Äì Best for data warehousing scenarios.

4. Specialized Indexes  
‚Ä¢ **Full-Text Index**  
  ‚Äì Supports word-based searches, ranking, inflections.  
  ‚Äì Vendor-specific commands (e.g. CREATE FULLTEXT INDEX).  
‚Ä¢ **Spatial Index**  
  ‚Äì Optimized for geometric data types (points, polygons).  
  ‚Äì Often R-tree or grid-based.  
‚Ä¢ **Filtered / Partial Index**  
  ‚Äì Indexes only rows matching a WHERE filter.  
  ‚Äì Reduces index size and improves speed for sparse predicates.  
  ‚Äì Example (PostgreSQL):  
    CREATE INDEX idx_active_users ON users(id) WHERE active = true;  
‚Ä¢ **Function-Based / Expression Index**  
  ‚Äì Indexes the result of an expression or function on a column.  
  ‚Äì Example (Oracle):  
    CREATE INDEX idx_upper_name ON employees(UPPER(last_name));

5. Creating & Dropping Indexes  
‚Ä¢ CREATE INDEX idx_name ON table_name(column1 [ASC|DESC], column2‚Ä¶);  
‚Ä¢ DROP INDEX idx_name [ON table_name];  

6. When to Use Indexes  
‚Ä¢ Columns in WHERE, JOIN, ORDER BY, GROUP BY clauses.  
‚Ä¢ High-cardinality columns (many distinct values).  
‚Ä¢ Foreign key columns to speed referential checks.  
‚Ä¢ Analytical queries on large fact tables‚Äîuse columnstore.

7. When to Avoid or Limit Indexes  
‚Ä¢ Small tables‚Äîfull scans are fast and indexes add overhead.  
‚Ä¢ Low-cardinality columns (e.g. gender flags).  
‚Ä¢ High-write tables‚Äîeach insert/update/delete must maintain indexes.  
‚Ä¢ Redundant indexes‚Äîavoid overlapping single-column and composite indexes.

8. Index Maintenance & Optimization  
‚Ä¢ **Rebuild/Reorganize** to defragment:  
  ‚Äì SQL Server: ALTER INDEX ALL ON table REBUILD;  
  ‚Äì Oracle: ALTER INDEX idx_name REBUILD;  
‚Ä¢ **Monitor Usage** via DMVs or system catalogs (e.g. sys.dm_db_index_usage_stats, pg_stat_user_indexes).  
‚Ä¢ **Adjust Fill Factor** for write-heavy tables to reduce page splits.  
‚Ä¢ **Drop Unused** indexes to improve write performance.

9. Covering Indexes  
‚Ä¢ Include non-key columns to satisfy queries entirely from the index‚Äîavoids lookups back to the table.  
‚Ä¢ Example (SQL Server):  
  CREATE INDEX idx_sales_cover ON sales(sale_date) INCLUDE(amount, region);

10. Monitoring & Tuning  
‚Ä¢ Use **EXPLAIN** or **EXPLAIN ANALYZE** to verify index usage vs. full scans.  
‚Ä¢ Check **fragmentation** or **bloat** and schedule regular maintenance.  
‚Ä¢ Leverage **filtered/partial** and **columnstore** indexes for targeted performance gains.

11. Summary Table

| Index Type              | Description                                                   |
|-------------------------|---------------------------------------------------------------|
| Clustered               | Rows ordered on disk by key; one per table                    |
| Non-Clustered           | Separate B-Tree with row pointers; many per table             |
| Heap                    | Table without clustered index; unordered rows                |
| Rowstore (B-Tree)       | Traditional row-by-row storage‚ÄîOLTP optimized                  |
| Columnstore             | Column-by-column storage‚ÄîOLAP optimized, highly compressed     |
| Clustered Columnstore   | Table stored entirely as columnstore                          |
| Full-Text               | Specialized word-search index                                 |
| Spatial                 | Geometric data index (R-tree/grid)                            |
| Filtered/Partial        | Index on subset of rows via filter predicate                  |
| Function-Based/Expr     | Index on an expression or function result                     |

Performance & Optimization for Data Analysts ‚Äì Indexing & Partitioning

1. Why It Matters  
   ‚Ä¢ Faster dashboards, reports, and ad-hoc queries‚Äîno more waiting minutes for results.  
   ‚Ä¢ Reduces server load and storage I/O, so more colleagues can run queries simultaneously.  
   ‚Ä¢ Enables handling of ever-growing data sets without starting from scratch.

‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ  
2. Indexing: Your Query ‚ÄúShortcut‚Äù  

A. What Is an Index?  
   ‚Ä¢ Think of it like the back of a book‚ÄîExcel doesn‚Äôt scan every row; it jumps to only relevant ones.  
   ‚Ä¢ Under the hood: B-Tree (rowstore) or compressed column segments (columnstore).

B. When to Use Rowstore (B-Tree) Indexes  
   ‚Ä¢ Point lookups and small ranges:  
     SQL: `WHERE customer_id = 123` or `sale_date BETWEEN '2025-01-01' AND '2025-01-31'`  
   ‚Ä¢ JOIN keys, foreign keys, GROUP BY and ORDER BY columns.  
   ‚Ä¢ Good default for OLTP and smaller fact tables.

C. When to Use Columnstore Indexes  
   ‚Ä¢ Scanning millions of rows for aggregates (SUM, AVG) across few columns.  
   ‚Ä¢ Data warehouse style queries: ‚ÄúTotal sales by region per quarter.‚Äù  
   ‚Ä¢ Highly compressed‚Äîcan reduce storage by 5‚Äì10√ó and run vectorized scans.

D. Designing Effective Indexes  
   1. **Selectivity**: index columns with many unique values first (e.g. transaction_id).  
   2. **Composite Index Order**: put the filter column most often used first:  
      - If you filter on `product_category` then `sale_date`, define `(product_category, sale_date)`.  
   3. **Covering Index**: add extra columns via INCLUDE (SQL Server) or INCLUDE clause (PostgreSQL) so your query never needs to ‚Äúlookup‚Äù back to the table.  
   4. **Filtered/Partial Index**: index only rows you care about (e.g. `WHERE is_active = true`) to keep index small and fast.

E. Maintenance & Monitoring  
   ‚Ä¢ **Fragmentation**: monitor with DMVs or system views‚Äîrebuild when >30% fragmented.  
   ‚Ä¢ **Statistics**: auto-update on, but refresh manually after big loads (`ANALYZE`, `UPDATE STATISTICS`).  
   ‚Ä¢ **Usage Metrics**: identify unused indexes (`sys.dm_db_index_usage_stats`, `pg_stat_user_indexes`) and drop them to speed up writes.

‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ  
3. Partitioning: Splitting Big Tables into ‚ÄúChunks‚Äù  

A. Why Partition?  
   ‚Ä¢ **Pruning**: queries with filters on the partition key (e.g. date) skip entire chunks‚Äîmassive I/O savings.  
   ‚Ä¢ **Maintenance**: you can archive, rebuild, or delete old partitions (e.g. last year‚Äôs data) without touching current data.  
   ‚Ä¢ **Parallelism**: some engines scan different partitions on different CPU threads.

B. Common Partitioning Schemes  
   1. **Range**: ideal for dates‚Äîe.g. one partition per month or quarter.  
   2. **List**: for categorical splits‚Äîe.g. one partition per region (‚ÄúNorth‚Äù, ‚ÄúSouth‚Äù, etc.).  
   3. **Hash**: distributes evenly when you can‚Äôt segment by date or list (e.g. user_id hash).  
   4. **Composite**: range + hash for high-volume date partitions that need sub-distribution.

C. Implementing Partitions (Conceptual)  
   ‚Ä¢ SQL Server: create a PARTITION FUNCTION (defines ranges) and a PARTITION SCHEME (maps to filegroups), then `CREATE TABLE ‚Ä¶ ON scheme`.  
   ‚Ä¢ Oracle/PostgreSQL/MySQL: use `PARTITION BY` clauses directly in `CREATE TABLE`.  

D. Partitioned Indexes  
   ‚Ä¢ **Local Index**: each partition has its own index‚Äîprunes together with the table.  
   ‚Ä¢ **Global Index**: a single index across all partitions‚Äîneeded if you query on non-partition columns, but cannot prune as efficiently.

E. Managing Partitions  
   ‚Ä¢ **SPLIT / MERGE**: adjust boundaries‚Äîadd new months or retire old ones.  
   ‚Ä¢ **SWITCH / EXCHANGE**: swap a data table in/out for fast archiving.  
   ‚Ä¢ **TRUNCATE PARTITION**: delete old data instantly (no logging).  
   ‚Ä¢ **Stats per Partition**: ensure the optimizer knows row counts per partition for accurate plans.

‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ  
4. Putting It Together‚ÄîA Data Analyst‚Äôs Checklist  
   1. **Analyze Your Workload**  
      ‚Äì Use `EXPLAIN`/`EXPLAIN ANALYZE` to see table scans vs. index seeks.  
      ‚Äì Profile slow queries in your BI tool or database logs.  
   2. **Design Indexes**  
      ‚Äì Start with columns in WHERE, JOIN, GROUP BY, ORDER BY.  
      ‚Äì Build rowstore B-Trees for selective filters; add columnstore for heavy aggregates.  
   3. **Partition Large Tables**  
      ‚Äì If your fact table exceeds tens of millions of rows, partition by date or region.  
      ‚Äì Test ‚Äúpartition pruning‚Äù by filtering on the key‚Äîensure only relevant partitions are scanned.  
   4. **Monitor & Maintain**  
      ‚Äì Schedule index rebuilds/reorganizes and statistics updates during off-peak hours.  
      ‚Äì Drop or disable unused indexes to speed up inserts/updates.  
      ‚Äì Adjust fill factors to balance page density vs. updates.  
   5. **Iterate**  
      ‚Äì Re-examine plans after major data volume changes (e.g. year-end loads).  
      ‚Äì Archive old partitions and merge low-value ones to keep management overhead low.
